PRÉ-FASE

Antes de escrever a primeira linha de código, você precisa entender como o pensamento lógico funciona. Um programador com lógica fraca vai sofrer em todo problema, mesmo que saiba a sintaxe de memória. Esta seção te ensina a pensar como um computador — passo a passo, sem ambiguidade.

Fluxograma é um diagrama que representa visualmente a lógica de um programa antes de escrever código. Você usa formas geométricas com significados específicos:

⬭ Oval — Início e Fim do programa
▭ Retângulo — Processo / instrução (ex: "some X com Y")
◇ Losango — Decisão / condição (ex: "X é maior que 0?")
▱ Paralelogramo — Entrada ou saída de dados
→ Setas — Fluxo de execução (o caminho que o programa segue)

Antes de programar qualquer coisa minimamente complexa, desenhe o fluxograma. Parece trabalho extra, mas economiza horas de debug.

Proposição é qualquer afirmação que pode ser verdadeira (V) ou falsa (F). Ex: "5 > 3" é V. "gato é um peixe" é F.

Os operadores lógicos combinam proposições:

E (AND / &&) — Ambas precisam ser V para o resultado ser V
  V e V = V  |  V e F = F  |  F e F = F

OU (OR / ||) — Pelo menos uma precisa ser V
  V ou V = V  |  V ou F = V  |  F ou F = F

NÃO (NOT / !) — Inverte o valor
  não V = F  |  não F = V

Isso vira condicionais no código: if (idade >= 18 && temCNH)

O computador só entende dois estados: ligado (1) e desligado (0). Tudo — números, textos, imagens, sons — é armazenado como sequências de 0s e 1s.

Como converter decimal para binário:
Divida por 2 sucessivamente e anote os restos de baixo para cima.
Ex: 13 → 13÷2=6 R1, 6÷2=3 R0, 3÷2=1 R1, 1÷2=0 R1 → binário: 1101

Como converter binário para decimal:
Multiplique cada bit pela potência de 2 correspondente.
Ex: 1101 = 1×8 + 1×4 + 0×2 + 1×1 = 8+4+0+1 = 13

Bit = 1 dígito binário (0 ou 1)
Byte = 8 bits (pode representar 256 valores: 0 a 255)
ASCII = padrão que mapeia bytes para caracteres (ex: 'A' = 65 = 01000001)

Quando você declara uma variável, o computador reserva um espaço na memória RAM e dá um nome a ele. Pense na RAM como uma planilha gigante: cada célula tem um endereço (número) e um valor.

Tipos e tamanho na memória:
• boolean — 1 bit (0 ou 1)
• int/number — 4 a 8 bytes
• string — 1 byte por caractere (ASCII) ou 2 bytes (Unicode)

Stack vs Heap:
• Stack — memória rápida, organizada, para valores simples (numbers, booleans)
• Heap — memória maior, para objetos e arrays (o JS guarda o endereço na stack, o valor no heap)

Entender isso explica por que let a = b copia o valor, mas let a = obj copia apenas a referência.

Algoritmo é uma sequência finita de passos bem definidos para resolver um problema. Todo programa de computador é um algoritmo. A diferença entre um programador bom e ruim muitas vezes está apenas em saber criar algoritmos eficientes.

Um bom algoritmo tem 3 características:
• Finito — termina em algum momento
• Determinístico — sempre dá o mesmo resultado para a mesma entrada
• Eficiente — resolve o problema sem desperdiçar tempo/memória

Exemplo prático: o algoritmo para fazer um sanduíche:
1. Pegue o pão
2. Abra o pão ao meio
3. Passe a maionese na parte de baixo
4. Coloque o presunto
5. Feche o pão

Parece óbvio, mas essa precisão exata é o que o computador precisa.

Recursão é quando uma função chama a si mesma para resolver uma versão menor do mesmo problema. É como se a função dividisse o problema em pedaços menores até chegar num caso tão simples que dá pra resolver direto — esse é o caso base. Sem caso base, a recursão nunca para.

Busca linear percorre o array do início ao fim — simples mas lento para arrays grandes. Busca binária divide o array ao meio a cada passo — muito mais rápida, mas exige que o array esteja ordenado. Big O mede como o tempo do algoritmo cresce conforme a entrada fica maior. Não é sobre tempo real — é sobre a taxa de crescimento. O(1) significa sempre o mesmo tempo. O(n) significa proporcional ao tamanho. O(n²) significa que se dobrar a entrada, o tempo quadruplica. Sempre busque o menor Big O possível.

BST é uma estrutura em que cada nó tem no máximo 2 filhos. A regra: esquerda sempre menor, direita sempre maior. Isso torna busca, inserção e remoção O(log n) — se a árvore estiver balanceada. É como um dicionário ordenado em estrutura de árvore.

Merge Sort divide o array ao meio, ordena cada metade recursivamente e as mescla — sempre O(n log n). Quick Sort escolhe um pivô, coloca menores à esquerda e maiores à direita recursivamente — O(n log n) na média. Ambos são infinitamente mais eficientes que Bubble Sort para arrays grandes.

// VÍDEO RECOMENDADO

Grafo é uma estrutura com nós (vértices) e conexões (arestas). BFS (Busca em Largura) usa uma Queue e visita nível por nível — ótimo para menor caminho. DFS (Busca em Profundidade) usa uma Stack (ou recursão) e vai fundo em cada caminho antes de voltar.

Express é o framework web mais popular do Node.js. Simplifica criação de rotas (URLs que a API responde) e middlewares (funções que rodam entre a requisição e a resposta). Middleware é um dos conceitos mais poderosos do Express.

REST é um conjunto de convenções para criar APIs HTTP. Os verbos têm significados: GET busca, POST cria, PUT/PATCH atualiza, DELETE remove. Os status codes comunicam o resultado: 200 OK, 201 criado, 400 erro do cliente, 401 não autenticado, 404 não encontrado, 500 erro do servidor.

Zod é uma biblioteca de validação de dados em tempo real. Você define um schema (regras) e valida qualquer dado contra ele. Se algo vier errado do front-end (campo em branco, tipo errado, email inválido), Zod rejeita antes de você tentar salvar no banco.

JWT é um token assinado digitalmente que prova a identidade do usuário sem precisar consultar o banco a cada requisição. Tem 3 partes: header (algoritmo), payload (dados do usuário) e signature (assinatura). Qualquer servidor com a chave secreta pode validar.

// VÍDEO RECOMENDADO

Nunca salve senha em texto puro. Bcrypt aplica hash com salt aleatório — mesmo que o banco vaze, as senhas ficam protegidas. Roles definem o que cada usuário pode fazer. Rate Limiting impede ataques de força bruta.

// 3 exercícios

// 05 — ARQUITETURA

Arquitetura de Projetos

O que separa código amador de código profissional. Entenda do zero.

O que é arquitetura de software?

Imagina que você vai construir uma casa. Você não começa colocando tijolos aleatoriamente — primeiro vem o projeto arquitetônico: onde fica cada cômodo, como eles se conectam, onde passam os canos e a fiação.

Arquitetura de software é exatamente isso para o código. É a organização de como as partes do sistema se dividem, o que cada parte faz e como elas se comunicam. Um código sem arquitetura funciona — mas só até um ponto. Quando o projeto cresce, vira uma bagunça impossível de manter.

Sem arquitetura, um projeto começa assim: você coloca tudo em um arquivo. Depois dois. Depois dez. Seis meses depois ninguém — nem você — sabe onde está cada coisa. Adicionar uma feature nova quebra três outras. Isso tem nome: código espaguete.

Com arquitetura, cada coisa tem seu lugar. O arquivo que define rotas não sabe nada do banco de dados. O arquivo de regras de negócio não sabe nada do HTTP. Separação de responsabilidades — cada parte faz uma coisa só, e bem.

Analogia simples: pensa numa cozinha profissional. Tem o chef que cria a receita, o cozinheiro que executa, o garçom que serve e o caixa que cobra. Cada um tem sua função. Se o chef fosse até a mesa servir e cobrar, seria um caos. Arquitetura é isso — cada camada tem seu papel.
📁

MVC — Model View Controller

// A mais usada em APIs REST — comece aqui
MVC divide o projeto em 3 camadas com responsabilidades bem definidas.

Model — são os dados. Define como os dados são estruturados, as regras de validação e como são salvos no banco.

View — em APIs REST, é a resposta JSON que você envia. Em apps com front-end, seria o HTML.

Controller — o intermediário. Recebe a requisição HTTP, chama o Model para buscar/salvar dados, e devolve a View.

Na prática moderna, adicionamos uma camada de Service entre Controller e Model para isolar regras de negócio. 
src/
├── controllers/        ← recebe req, chama service
│   └── user.controller.js
├── services/           ← regras de negócio
│   └── user.service.js
├── repositories/       ← fala com o banco
│   └── user.repository.js
├── routes/             ← define as URLs
│   └── user.routes.js
├── middlewares/        ← auth, log, validação
│   └── auth.middleware.js
└── app.js              ← junta tudo
Regra de ouro: Controller nunca acessa o banco diretamente. Passa pelo Service, que passa pelo Repository. Assim você pode trocar o banco sem tocar no Controller.
🧩

Feature-Based (por módulo)

// Melhor para projetos com muitas features
Em vez de agrupar por tipo (todos os controllers juntos, todos os services juntos), você agrupa por feature. Tudo relacionado a "auth" fica numa pasta, tudo de "products" em outra.

Vantagem enorme: se você precisar deletar a feature de produtos do projeto, é só deletar a pasta. Nada mais vai quebrar. Em MVC puro, você teria que caçar arquivos em 5 pastas diferentes. 
src/
├── features/
│   ├── auth/
│   │   ├── auth.controller.js
│   │   ├── auth.service.js
│   │   ├── auth.routes.js
│   │   └── auth.schema.js
│   ├── products/
│   │   ├── products.controller.js
│   │   └── ...
│   └── orders/
│       └── ...
├── shared/
│   ├── middlewares/
│   └── utils/
└── app.js
Use quando: o projeto tem mais de 3–4 features independentes. Escala bem em equipes grandes — times diferentes cuidam de features diferentes sem conflito.
🧅

Clean Architecture

// Para projetos complexos e altamente testáveis
Criada pelo Uncle Bob (mesmo do Clean Code). A ideia central é que as regras de negócio não dependem de nada externo — nem do banco, nem do framework, nem do HTTP.

Imagine camadas como cebola: no centro ficam as regras de negócio (entities e use cases). As camadas externas (banco, Express) dependem do centro — nunca o contrário.

Resultado: você pode trocar PostgreSQL por MongoDB, ou Express por Fastify, sem tocar nas regras de negócio. E testar as regras sem precisar subir o servidor. 
src/
├── domain/
│   ├── entities/     ← classes de negócio puras
│   └── usecases/     ← regras (sem banco, sem HTTP)
├── application/
│   └── interfaces/   ← contratos/interfaces
├── infrastructure/
│   ├── database/     ← Prisma, Mongoose
│   └── http/         ← Express, rotas
└── main/
    └── server.js     ← monta tudo
Atenção: Clean Architecture tem uma curva de aprendizado grande. Domine MVC primeiro. Volte aqui quando for construir um projeto sério.
⚡

Microservices

// Para times grandes e sistemas complexos
Microservices divide o sistema em serviços pequenos e independentes, cada um com sua responsabilidade específica e seu próprio banco de dados.

Comunicação: os serviços conversam via HTTP (REST ou gRPC) ou via mensageria (RabbitMQ, Kafka). Um API Gateway recebe todas as requisições de fora e distribui para o serviço certo.

Importante: microservices resolve problemas de escala de equipe e de tráfego. Se você está começando, comece com monolito (MVC ou Feature-Based). Migre para micro quando precisar. 
services/
├── api-gateway/       ← porta 3000 (entrada)
├── auth-service/      ← porta 3001
├── product-service/   ← porta 3002
├── order-service/     ← porta 3003
└── docker-compose.yml ← sobe todos
Regra prática: "Faça um monolito até doer." Só migre para microservices quando o monolito virar um problema real de escala ou equipe.
// SOLID — OS 5 PRINCÍPIOS DO CÓDIGO PROFISSIONAL

O que é SOLID?

SOLID são 5 princípios de design de código criados para tornar software fácil de manter, estender e testar. Não são regras absolutas — são guias. Quando você sentir que adicionar uma feature nova quebra várias coisas, provavelmente está violando algum princípio SOLID.

S

Single Responsibility

Cada classe ou função deve ter apenas uma razão para mudar. Se sua classe faz duas coisas, separe em duas classes.

❌ Ruim: classe UserService que valida dados, salva no banco E envia email.
✅ Bom: UserService (salva), UserValidator (valida), EmailService (envia).
O

Open/Closed

Aberto para extensão, fechado para modificação. Adicione funcionalidades sem modificar o código existente.

❌ Ruim: if (tipo === 'email') { ... } else if (tipo === 'sms') { ... }
✅ Bom: cada notificação implementa interface Notificacao.enviar().
L

Liskov Substitution

Uma subclasse deve substituir a classe pai sem quebrar o comportamento esperado do programa.

❌ Ruim: Pinguim extends Ave e lança erro no método voar().
✅ Bom: separar AvesVoadoras de AvesNaoVoadoras.
I

Interface Segregation

Não force uma classe a implementar métodos que ela não usa. Interfaces pequenas e específicas são melhores.

❌ Ruim: interface Animal com voar(), nadar() e correr() para todos os animais.
✅ Bom: interfaces separadas: Voador, Nadador, Corredor.
D

Dependency Inversion

Dependa de abstrações, não de implementações concretas. Facilita troca de dependências e testes.

❌ Ruim: UserService cria new PrismaRepository() diretamente.
✅ Bom: UserService recebe um repositório por parâmetro (injeção de dependência).
💡 Ordem de aprendizado: MVC primeiro (aplique nos projetos da Fase 4) → Feature-Based (aplique nos projetos maiores) → SOLID (aplique sempre, em qualquer código) → Clean Architecture (projetos sérios) → Microservices (quando o monolito doer).

// 11 — MENTALIDADE

Regras Para Crescer de Verdade

O código é a parte fácil. A mentalidade é o que separa quem chega lá de quem desiste.

// REGRA #1

Construa. Não só assista.

Para cada 30 min de vídeo → 1h codando. Tutorial sem prática não ensina nada.

// REGRA #2

GitHub todo dia.

Mesmo que seja 1 linha. Seu histórico de commits é seu portfólio real.

// REGRA #3

Leia código de outros.

Abra projetos open source e leia. Aprende o que nenhum tutorial ensina.

// REGRA #4

Inglês desde agora.

Leia a MDN em inglês hoje. 90% dos melhores recursos são em inglês.

// REGRA #5

Bug é parte do trabalho.

Leia o erro, pesquise, entenda — não só copie a solução. É assim que se aprende.

// REGRA #6

Consistência vence maratona.

1 hora por dia todo dia vale mais que 8h no fim de semana. Apareça todo dia.

Feito pra você ir longe. // Bora codar 🚀
